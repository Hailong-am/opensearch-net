# Design: `combined_fields` query support

**Issue:** [opensearch-net#956](https://github.com/opensearch-project/opensearch-net/issues/956)
**Date:** 2026-07-08

## Problem

OpenSearch v3.2 added the `combined_fields` full-text query, which searches multiple
`text` fields as though their terms were combined into one field, using a term-centric
BM25F scoring model. The `OpenSearch.Client` (NEST-derived) high-level client does not
expose this query type, so users cannot build a `combined_fields` query in a strongly
typed way.

## Goal

Add a `CombinedFieldsQuery` full-text query type to `OpenSearch.Client`, exposed as
`combined_fields` inside a `QueryContainer`, supporting both the object-initializer and
fluent (descriptor) construction styles, with serialization/deserialization and the
usual visitor/pretty-print wiring — following the existing query-type conventions.

## Non-goals

- No changes to the API specification — `combined_fields` is already defined in
  `opensearch-api-specification` (`spec/schemas/_common.query_dsl.yaml`:
  `CombinedFieldsQuery`, `CombinedFieldsOperator`, `CombinedFieldsZeroTerms`, and the
  `combined_fields` entry in the query container).
- No changes to generated code (`src/**/_Generated/`) or the `ApiGenerator`. Query DSL
  types are hand-written; the generator only produces endpoints/requests/descriptors.
- No new low-level (`OpenSearch.Net`) changes — this is a high-level query type only.

## Background: why hand-written, and why a dedicated type

**Hand-written.** The `ApiGenerator` (`./build.sh codegen`) reads the OpenAPI spec and
emits only endpoint plumbing into `_Generated/` (Requests, Descriptors, client methods,
request parameters, URL lookups). The Query DSL types live outside `_Generated/` and
carry no `<auto-generated>` marker — they are maintained by hand. So `combined_fields`
must be written by hand, modeled on the closest existing analog, `MultiMatchQuery`.

**Dedicated type (not reusing `MultiMatchQuery`).** `combined_fields` is a *standalone*
query type in the spec — it is NOT a value of `TextQueryType` (the `multi_match.type`
enum). Its fields are a strict subset (6) of MultiMatch's (18):

| CombinedFields field | MultiMatch has it? |
|---|---|
| `fields` | yes |
| `query` | yes |
| `operator` | yes |
| `minimum_should_match` | yes |
| `auto_generate_synonyms_phrase_query` | yes |
| `zero_terms_query` | yes |
| (12 others: `analyzer`, `fuzziness`, `tie_breaker`, `type`, `slop`, …) | MultiMatch-only |

Reusing the MultiMatch type is rejected because:

1. **Serialization identity.** `QueryContainerConverter` selects the output key
   (`multi_match` vs `combined_fields`) from which container property is set and the
   concrete type. Two query kinds sharing one type/property cannot be distinguished on
   write and collapse into one on read. Each query kind needs its own container property
   and `InternalWrapInContainer` target.
2. **Type safety.** Reusing MultiMatch would expose `analyzer`, `fuzziness`,
   `tie_breaker`, `type`, `slop`, etc. — none valid for `combined_fields`. A dedicated
   type exposes only the 6 valid knobs.
3. **Codebase convention.** Every query is its own interface/class/descriptor even when
   fields overlap heavily (e.g. `Match`, `MatchPhrase`, `MatchBoolPrefix` all repeat
   `query`/`analyzer`/`zero_terms_query` with no shared base). A shared base solely for
   `combined_fields` would break the established pattern and complicate wiring.

**Reuse happens at the value-type level:** `Fields`, `MinimumShouldMatch`, and the
existing `Operator` (`and`/`or`) and `ZeroTermsQuery` (`all`/`none`) enums — both exact
value matches for `CombinedFieldsOperator` and `CombinedFieldsZeroTerms`. No new enums.

## Query shape

| Property (C#) | JSON | Type | Required |
|---|---|---|---|
| `Fields` | `fields` | `Fields` | yes |
| `Query` | `query` | `string` | yes |
| `Operator` | `operator` | `Operator` (`and`/`or`) | no |
| `MinimumShouldMatch` | `minimum_should_match` | `MinimumShouldMatch` | no |
| `AutoGenerateSynonymsPhraseQuery` | `auto_generate_synonyms_phrase_query` | `bool?` | no |
| `ZeroTermsQuery` | `zero_terms_query` | `ZeroTermsQuery` (`all`/`none`) | no |

**Conditionless rule:** the query is conditionless when `Query` is null/empty. This
mirrors `MultiMatchQuery` (which also carries a `Fields` property but keys conditionless
solely off `Query`); the `Fields` type has no `HasValue()` helper and the codebase does
not guard on it here.

## Components and files

New folder: `src/OpenSearch.Client/QueryDsl/FullText/CombinedFields/`

1. **`CombinedFieldsQuery.cs`** — three members, mirroring `MultiMatchQuery.cs`:
   - `ICombinedFieldsQuery : IQuery` — `[InterfaceDataContract]`,
     `[ReadAs(typeof(CombinedFieldsQuery))]`, one `[DataMember(Name = "...")]` per field.
   - `CombinedFieldsQuery : QueryBase, ICombinedFieldsQuery` —
     `protected override bool Conditionless => IsConditionless(this);`
     `internal override void InternalWrapInContainer(IQueryContainer c) => c.CombinedFields = this;`
     `internal static bool IsConditionless(ICombinedFieldsQuery q) => q.Query.IsNullOrEmpty() || !q.Fields.HasValue();`
   - `CombinedFieldsQueryDescriptor<T> : QueryDescriptorBase<…, ICombinedFieldsQuery>, ICombinedFieldsQuery where T : class`
     — explicit interface impls + fluent `Assign(...)` setters.

Wiring (edits to existing files):

2. `QueryDsl/Abstractions/Container/IQueryContainer.cs` — add
   `[DataMember(Name = "combined_fields")] ICombinedFieldsQuery CombinedFields { get; set; }`
3. `QueryDsl/Abstractions/Container/QueryContainer-Assignments.cs` — backing field
   `_combinedFields` + explicit property `ICombinedFieldsQuery IQueryContainer.CombinedFields`.
   Serialization/deserialization needs no converter changes: `QueryContainerFormatter`
   serializes via reflection over `IQueryContainer`'s `[DataMember]` properties
   (`DynamicObjectResolver`), and deserialization is driven by the `[ReadAs]` attribute
   on `ICombinedFieldsQuery`. The container property added above is sufficient for both.
5. `QueryDsl/Abstractions/Container/QueryContainerDescriptor.cs` — fluent entry
   `public QueryContainer CombinedFields(Func<CombinedFieldsQueryDescriptor<T>, ICombinedFieldsQuery> selector) => WrapInContainer(selector, (q, c) => c.CombinedFields = q);`
6. `QueryDsl/Query.cs` — static `Query<T>.CombinedFields(...)` helper.
7. `QueryDsl/Visitor/QueryVisitor.cs` — `IQueryVisitor.Visit(ICombinedFieldsQuery)` +
   `QueryVisitor` base virtual impl.
8. `QueryDsl/Visitor/QueryWalker.cs` — `VisitQuery(qd.CombinedFields, visitor, …)`.
9. `QueryDsl/Visitor/DslPrettyPrintVisitor.cs` —
   `Visit(ICombinedFieldsQuery) => Write("combined_fields")`.

No new enums; no generated-code changes.

## Data flow

- **Serialize:** `QueryContainer.CombinedFields` set → `QueryContainerConverter` write
  branch emits `"combined_fields": { … }`.
- **Deserialize:** `"combined_fields"` key → converter read case → `CombinedFieldsQuery`
  via `[ReadAs]`.
- **Conditionless:** empty query/fields → query omitted from the container (standard
  behavior via `QueryBase`/`Conditionless`).

## Testing

Follow the `MultiMatchUsageTests` pattern under
`tests/Tests/QueryDsl/FullText/CombinedFields/CombinedFieldsUsageTests.cs`:

- Subclass `QueryDslUsageTestsBase`.
- `ConditionlessWhen` via `new ConditionlessWhen<ICombinedFieldsQuery>(a => a.CombinedFields) { … }`.
- `QueryInitializer` (object-initializer form) + `QueryFluent` (`.CombinedFields(c => c…)`).
- Anonymous `combined_fields = new { … }` expected-JSON block asserting all six fields
  serialize under the correct names.
- A boost/variant class mirroring `MultiMatchQueryWithBoost` to cover `QueryBase`
  common fields (`boost`, `_name`).

## Error handling

No new error paths. Invalid combinations (e.g. `type`-style options) are structurally
impossible because the type does not expose them. Server-side validation of field types
(`text` only) remains the server's responsibility, consistent with other query types.
